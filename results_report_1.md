# 📑 NYC Taxi Prediction: Experimental Results & Analysis (PRACTICE 1)

## 1. Executive Summary
This report analyzes the performance of simplified machine learning pipelines for the **NYC Green Taxi dataset**. Two tasks were performed: **Tip Regression** and **Payment Type Classification**.

The experiments were incrementally conducted by adding features to an XGBoost model. The summary of the results is as follows:
*   **Regression:** Adding features resulted in **statistically significant improvements**.
*   **Classification:** Adding features improved metrics marginally, but **failed statistical significance tests**, suggesting the simpler model is preferable.

## 2. Methodology Overview
To ensure scientific rigor, the following pipeline strategy was applied to all experiments:

1.  **Fixed Test Set:** A 20% hold-out set with random seed (`42`).
2.  **Cross-Validation (CV) for Tuning:** With only the training set, 5-Fold CV was used strictly to determine the **optimal number of boosting rounds** (epochs) via early stopping.
3.  **Retraining Strategy:** The average optimal epoch count from CV was calculated. The model was then retrained on the **full 80% training set** using this specific epoch count. ℹ️ NOTE! This strategy was used for simplicity, but the std between runs was high, suggesting the need for more conservative/robust approaches.
4.  **Statistical Testing:** Not rely solely on higher numbers:
    *   **Wilcoxon Signed-Rank Test** for Regression (comparing error magnitudes).
    *   **McNemar’s Test** for Classification (comparing agreement/disagreement vectors).

**Conclusion:** For the regression task, the complexity of adding Location IDs is justified by a statistically proven reduction in error.

## 5. Critical Analysis & Risk Assessment (**PRACTICE QUESTIONS**)

### A. How can you ensure that the improvement is not due to randomness or data leakage?

1.  **Addressing Randomness:**
    *   **Fixed Seeds:** `RANDOM_SEED=42` for splits and model initialization. This guarantees that if we re-run the code, we get the exact same results.
    *   **Statistical Tests:** This is the most important factor. In Classification, the metrics *looked* better for the 5-feature model, but McNemar's test revealed it was noise (p > 0.05). In Regression, the p-values were infinitesimal ($10^{-50}$), proving the improvement was genuine and not random fluctuation.

2.  **Addressing Data Leakage:**
    *   **Strict Split Separation:** The `train_test_split` happens **before** any logic in the pipeline. The Test set is never seen by the Cross-Validation loop.
    *   **Target Transformation:** We transformed targets (shifting classes 1-4 to 0-2) individually on vectors, ensuring no aggregate statistics from the test set were used to map the training set.
    *   **Feature Choice:** We avoided features that are only known *after* the trip concludes (like `tip_amount` for the classification task or `total_amount` which includes the tip).

### B. What risks exist if a feature cannot be reliably generated in production?

In the Regression task, `DOLocationID` (Drop-off Location) provided significant lift. However, relying on it creates specific risks:

1.  **Availability:** In a real-time taxi app, the user inputs a destination, but the driver might drop them off elsewhere due to many 'real-life' reasons, or the GPS signal might fail/turn-off. If `DOLocationID` is missing at the moment of prediction (inference), the model (trained on 5 features) will crash or require a complex fallback mechanism.
2.  **Drift:** If the `DOLocationID` definitions change (e.g., Local map or taxi APP re-zones the map regions), the integer IDs fed into the model will no longer correspond to the geographic reality it learned, degrading performance instantly.
3.  **User Friction:** If the model *requires* a Drop-off location to predict the tip, the app cannot give an estimate until the user enters a destination. If the business requirement changes to "predict tip before destination entry," the Champion model becomes useless.

**Recommendations:** 

- For Regression:
    1. keep the 5-feature model but build a fallback to the 3-feature model if location services fail.
    2. Try to change the location service data storaging to geographic coordinates. They don't risk changing as the IDs. These coords could be grouped by a deterministic distance clustering algorithm equivalent to the integer zones.
    3. When trip type is 'street-hail', if the location service is not available, apply a fallback solution with a model trained without these features. For Dispatch trip on app, pick the locations during the trip order, despite possible changes.
- For Classification, deploy the 3-feature model to minimize data dependency risks since additional features added no value.
- In summary, each task type and feature has its own conjecture that should be carefully analyzed and prepared for exceptions. (e.g., it was not the current case, but usually it's possible to prepare automatic data imputation in some features)
